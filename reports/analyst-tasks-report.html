Eddard's top-line notes for the refactor

  • Four checks are dead code, for two different reasons. SecurityPrimaryExchangeCheck is fully implemented and has a passing spec, but is never called from AnalystTaskRunner or anywhere else - grepping the entire app/lib/spec tree turns up only its own file and its own test. Separately, three ETF-constituent checks (EtfConstituentsCoveredCheck, EtfConstituentsLastDividendCheck, EtfConstituentsMatchedCheck) are deliberately commented out in the runner. Both are cleanup candidates, but they need different treatment: one looks like an accidental omission, the other looks like an intentional (but undocumented) disabling - worth asking the team which is which before deleting anything.
  • There is no check registry - a new check only runs if someone remembers to add one line to a 55-line hardcoded method. AnalystTaskRunner#perform_all_tasks (app/tasks/analyst_task_runner.rb:49-104) is a flat, manually-maintained list of .perform_now calls. This is almost certainly how SecurityPrimaryExchangeCheck above ended up orphaned. A refactor should replace this with an explicit registry (even a simple array of check classes) so "written and tested" and "actually runs" can't silently diverge again.
  • The analyst-facing "resolve" button lies about success. AnalystTasksController#destroy (app/controllers/admin/analyst_tasks_controller.rb:22-31) only calls complete! when can_be_thumbsed_up? is true, but always responds {success: true} regardless. For the ~35 of 58 task types that aren't dismissible, an analyst clicking "resolve" gets a false green light with no explanation of why the task is still there. This is a real, narrowly-scoped bug independent of any larger refactor - worth fixing on its own.
  • Two independent checks police "dividends too close together" and can disagree. CloseDividendsCheck and QuarterliesLessThanTwoMonthsCheck overlap for quarterly dividends but use different formulas (a raw day-count cutoff vs. a rounded month conversion) and raise different task types - a 40-day gap trips one but not the other (see both cards for the worked example). Worth consolidating into one rule with one threshold.
  • Outstanding task IDs are not stable across runs. Every check run deletes and recreates its own outstanding tasks from scratch (app/tasks/abstract_task.rb:9) rather than updating existing rows. Any bookmark, deep link, or external reference to a specific outstanding task's ID can silently point at a deleted row after the next scheduled run recreates the same logical task under a new ID. Processed (resolved) tasks are untouched by this, since the delete only targets outstanding ones.
