Eddard's top-line notes for the refactor

  • This is a bespoke rule DSL with no spec. DateRule (app/models/date_rule.rb) defines 19 value_type integer constants that combine into rulesets. The only "documentation" of what a valid combination means lives implicitly across RuleBuilder, DateForecaster, and RuleFormatter - three files that each independently encode which value types are legal together. A refactor should extract one canonical rule-shape definition (even just a lookup table of valid combinations) that all three read from, instead of three parallel case statements that can silently drift out of sync.
  • Country/index exceptions are scattered, not centralized. Japan, Korea, and FTSE MIB constituents each get hardcoded rule overrides, but the checks live in different files (RuleBuilder, DateForecaster) using different mechanisms (bucket-level hardcoding vs. forecast-time branching). A strategy-pattern registry keyed by country/index would make it obvious where to add the next exception, and obvious what exceptions already exist.
  • Confidence scoring is tunable but untested at the boundary. The calculate_confidence formula (rule #20) and the COVID-19 carve-out (rule #21) are hand-tuned magic numbers with no named constants. A single dividend landing right at the 50% "sufficient confidence" cliff-edge flips the forecaster between a learned pattern and a naive guess - worth adding explicit boundary tests before touching this formula.
  • No DB-level guardrails on date ordering. Nothing enforces declaration_date ≤ ex_div_date ≤ record_date ≤ pay_date at the model or database level (see rule #47) - the only place that ordering is *approximately* enforced is the pay-date-after-record-date check in DateCalculator (rule #43), and even that runs imperatively, not as a validation. Consider adding this as an explicit, testable model validation independent of the forecasting engine.
  • The holiday calendar has no recurrence rule (rule #33) - every public holiday for every country is a manually inserted row for a single exact date. If this table isn't kept current, business-day math silently degrades (a missed holiday just looks like an ordinary work day) rather than failing loudly. Worth a health-check job that flags countries with no holidays on file for the upcoming year.
